Supervisor 管理进程

发表于 2017-12-17

简介

Supervisor是客户端/服务端系统，使得用户能够控制Unix类操作系统上的进程，有这几点值得肯定:

方便

普通情况下一般都是写rc.d脚本文件来控制进程实例。rc.d文件用来控制进程的初始化、自动启动和管理，但是书写和管理这些脚本并不容易，而且，rc.d脚本并不能自动重启crash掉的进程。Supervisor 将进程作为其子进程来启动，所以能够做到crash的进程能够自动重启。

精确

通常在Unix系统中很难获得进程的 up/down 状态。Supervisor 将进程作为子进程来启动，这样很容易获得其运行的状态。

特性

简单，中心化，高效，可扩展，可移植，值得信赖

Supervisor 组件

supervisord

服务端组件，负责启动子程序，返回来自客户端的命令，重启 crash 或者是退出的子进程，记录进程日志——stdout和stderr的输出，子进程生命周期事件的产生和处理。

supervisorctl

客户端命令行工具，是supervisord提供的shell类的接口。通过supervisorctl，用户可以连接到不同的supervisord进程，获得所控制的子进程的状态，启动和停止子进程，获得supervisord运行进程的列表。
命令行客户端和服务端进程通过Unix域socket或者是TCP socket通信。配置文件段使用[supervisorctl]。

web server

supervisord暴漏的web客户端，通过访问服务端URL：http://localhost:9001 来查看和控制进程状态。配置文件位于[inet_http_server]段。

微服务简介(译)

发表于 2017-06-28 | 分类于 Microservices

前言

微服务架构模式反复的在各大架构会议中提到，其热度可见一斑。各大互联网企业已经慢慢的将系统架构朝微服务发展，究竟什么是微服务架构？它和完全独立统一部署的的架构(monolithic)和 SOA 架构有什么区别？

原型的描述

无论实现风格是怎样的，第一个核心的概念就是 独立部署单元，每一个微服务架构的组件都是作为独立的部署单元，这样满足高效的持续交付，扩展能力强，而且业务之间相互解耦。

最重要的概念是 服务单元 （ service component ）。服务组件由一个或者多个模块组成，其包含一组功能明确的功能，或者是大型业务应用的独立组成部分。设计服务组件粒度的大小是微服务架构最具有挑战的工作。

微服务架构模式中另外一个核心概念是 分布式 架构，意味着所有的微服务架构中的组件完全同其他组件解耦，他们之间通过远程访问协议通讯（ 如：JMS, AMQP, REST, SOAP, RMI 等 ）。分布式天然的获得了超强的水平扩展能力和易部署的特性。

微服务架构是从其他架构中进化而来的，并不是什么全新的架构。主要来自于：独立统一部署的的架构(monolithic)的分层架构模式和分布式应用的 SOA 架构模式。微服务解决上述两种架构的问题：1. 独立统一部署的架构的持续交付和扩展性差等问题；2. SOA 架构模式的并非真正的分布式和去中心化问题，微服务架构服务单元间弱依赖和轻通信，SOA 架构过于复杂，而且成本高昂。微服务架构模式可以理解为清爽型 SOA 架构模式。

模式拓扑结构

有很多种微服务实现方式，大部分可以总结为 基于REST的API 方式、基于REST的应用方式、和消息中心化。

基于REST的API 方式 用来给网页提供微型的自包含的独立服务。如下图所示，整个拓扑结构由粒度很小的服务组件构成。通常只由包含特定的业务逻辑的一到两个服务模块构成。这种拓扑模式下，细粒度的服务访问都使用 REST 接口方式，通常使用单独部署的 web API 层来实现。

基于REST的应用方式 和 基于REST的API 方式 不同之处在于客户端请求都是接受来自于传统 web 应用或者是胖客户端应用。如图所示，UI 层也是独立部署的，但是服务组件的粒度稍大一些，包含全部业务逻辑的一小部分。这种方式在小型到中型业务应用中比较常见，服务模块间的依赖复杂度相对较低。

消息中心化（ 可能还有 RPC 方式如 Thrift, gRPC 等），和 基于REST的应用方式 类似，只是在 UI 层和服务模块之间增加了一层轻量级的消息中间件（如：ActiveMQ, ZeroMQ, 等）。

避免依赖和复杂的服务编排

微服务架构最大的挑战就是决定 服务模块的粒度大小。粒度太大，微服务架构就和 SOA 架构相比没有更大的优势；粒度太小，服务间依赖的编排复杂度就会急剧上升。一旦发现需要在 UI 层或者 API 层中编排服务组件，就意味着微服务粒度过小；如果发现模块中间存在服务调用，也意味着模块的粒度过小，无法做到业务功能独立化。

需要注意的地方

  1. 服务部署期间 API 版本的控制和流量控制
  2. 接口契约问题、服务治理、远程访问的权限问题（ OAuth 可解决）等。
  3. 性能问题，服务间通信是要成本的
  4. 系统集成测试
  5. 服务监控

为什么有 Java 8 ?

多核处理器的出现，其他流行的编程语言的影响；同时环境给了语言压力，促使语言的进化。Java 8 带给程序员的是提供了更丰富、更有效率的编程范式，以前遇到的问题可以用更优雅的方式解决，如流式处理 (Stream processing)、函数式编程 ( functional programming)。

Lambda 表达式

Lambda 表达式可以分解成这几个概念：

  1. 匿名：Lambda 表达式没有一个显式的名称
  2. 函数：Lambda 表达式不和任何一个方法相关联，但是它和函数一样的是具有参数列表、函数体和返回值，可能还会有异常抛出。
  3. 可传递: Lambda 表达式可以作为参数传递给方法或者存储在变量中。
  4. 简洁：不需要写大量的辅助代码，如匿名类中的成员方法名。

optional 类

写过 Java 的人对 NPE 异常一定不会陌生，为了避免该情况的发生，代码里面充斥着大量的 if/else 判断逻辑，着实不优雅。为了解决这个让无数程序员蛋疼的事情，第三方库如 Guava 或者是 JVM 类的语言Groovy,Scala 对 null 的问题都做了相应的处理。Java 8 里面也针对这类问题，估计是参考 Guava 的实现，引入了 Optional 类。Optional类的引入让我们对可为 null 的对象的处理用另外的思路去解决类似的问题，一味的使用判断逻辑来处理可能为 null 的对象着实已经过时。

javadoc

一个包含的值可能是空或者不为空的容器对象。如果值存在(isPresent())，返回 true，通过 get() 方法返回该值。

用法

of(T value)

返回 Optinal 对象，包含非空的 value 值，如果 value 值为空，会返回NullPointerException。

ofNullable(T value)

返回 Optional 对象，包含可能为空的 value 值。

get()

若 Optional 对象包含的值不为空，则返回该对象；如果为空，抛出NoSuchElementException异常

isPresent()

若 Optional 对象包含的不为空值返回 true，否则返回 false.

ifPresent(Consumer<? super T> consumer)

如果值存在，调用针对该值对应的 consumer 方法，反之不调用。

filter(Predicate<? super T> predicate)

如果值存在, 而且值匹配后面的预测条件，则返回包含该值的Optional 对象，反之返回空的 Optional 对象。

map(Function<? super T, ? extends U> mapper)

如果值存在, 调用后面的 map 函数方法， 如果结果为非 null, 则返回包含该值的Optional 对象，反之返回空的 Optional 对象。

flatMap(Function<? super T, Optional<U>> mapper)

和map类似，只是 mapper 的返回值也是Optional对象，如果返回的话，无需额外的使用Optional对象包裹。

orElse(T other)

如果Optional内值存在，则返回该值，如果值不存在则返回other对象

orElseGet(Supplier<? extends T> other)

如果Optional内值存在，则返回该值，如果值不存在则返回other 方法产生的对象

orElseThrow(Supplier<? extends X> exceptionSupplier) throws X

如果Optional内值存在，则返回该值，如果值不存在则抛出other 方法产生的异常。

MySQL 开发要点

发表于 2017-04-12 | 分类于 MySQL

前言

本文仅仅是用来做日常开发备忘，如果读者想深入了解 MySQL 高性能开发的完整内容，请阅读《High Performance MySQL》

表设计

  • 表必须定义主键，默认为 ID，整型自增
  • 禁止使用外键，使用外键性能和扩展性会随着业务增长大打折扣，一些开源项目中使用表设计需要针对这些进行调整
  • 字符集优先选择 UTF-8，避免后面国际化带来的迁移问题
  • 必须添加 gmt_create, gmt_modified 字段标记创建时间和修改时间
  • 超过 500万行或者容量超过 10G 考虑分表设计
  • 单条记录大小禁止超过 8K

字段设计

  • 索引列为 not null，并设置 default值。
  • 禁止使用 float,double 类型， 使用 decimal, int 替代
  • 禁止使用 blob, text 存储图片，文件等数据

其他

  • 禁止使用存储过程、触发器、函数、视图、时间等 MySQL 高级功能
  • 禁止使用跨库查询
  • 避免使用子查询，最好使用关联查询
  • 语句尽量避免使用大量的计算操作、多表关联、表遍历等操作
  • varchar长度避免过长，需要根据业务合理设置
  • 配置慢查询日志，做好监控

索引优化

去除重复的索引

重复索引有两个副作用

  1. 所有的 DML 语句将变得比较慢
  2. 数据库的磁盘占用量将会更大，备份恢复需要更长时间

可以使用工具检查是否使用重复索引，工具可 Google 下

删除不用的索引

可使用监控工具来监控表的索引数据的使用情况，定期将无用的索引移除掉，提升 DML 的速度以及减少磁盘的占用大小。

索引列的优化

使用合适的数据类型

  1. 主键使用 INT 而不是用 BIGINT，减少因索引带来的磁盘占用
  2. TIMESTAMP 类型占用4字节，DATETIME 类型占用8字节，根据情况来选择，如果没有特殊情况，使用 TIMESTAMP 类型
  3. 最好将列设置为 NOT NULL，尤其是初始设计时。

简化 SQL 语句

  1. 精简返回字段，能少返回的就少返回
  2. 谨慎使用子查询

InnoDB 参数

  • innodb_buffer_pool_size 比较重要的参数

用来缓存索引和行数据的，设置越大磁盘 IO 越少，也不能过大，防止频繁的 swap 调度，消耗过多的资源，一般设置为物理内存的 50%~80% 左右。

  • innodb_additional_mem_pool

存放 InnoDB 的内部目录，除了缓存表数据和索引外, 可以为操作所需的其他内部项分配缓存来提升 InnoDB 的性能，系统会自动调整，一般设置为 8M，表比较多可以适当增大。

  • innodb_log_file_size

指定日志的大小， 可设置为 4G 以下

  • innodb_log_buffer_size

事务在内存中缓存，控制在 2-8M，可适当增大

  • innodb_file_per_table

使每个Innodb的表，有自已独立的表空间。如删除文件后可以回收那部分空间。一般设置为1，对清理数据比较方便。

  • innodb_flush_method
    linux 设置为 O_DIRECT 直接写磁盘。

  • innodb_flush_neighbors
    SSD 上设置为0，顺序写对 SSD 没有提升

阻塞 IO 模型

简单的描述为：应用程序调用读函数，内核把数据准备好，复制到用户空间，然后返回给应用程序进行下一步处理。

在某些低速系统调用是可能使进程永远阻塞。

非阻塞 IO 模型

大家都不希望进程在 IO 操作不能完成的时候睡眠，而是希望系统调用立刻返回一个错误而返回，表示该操作如继续执行将阻塞。进程后面在合适的时候重新请求这个 IO 操作。这就是所谓的非阻塞的 IO 模型。基本思想就是：对文件描述符发出一个 read 操作。如果该输入上有数据，则读数据并返回它。如果无数据可读而报错，则立刻返回，并等待一段时间再去读取。整个过程称之为轮询请求。非阻塞 IO 模型不足之处是消耗了大量的 CPU 时间。一般情况下不会使用这种方法。

IO 多路转接（复用）

IO 多路转接（multiplexing）指的是在单个线程里通过记录跟踪每个 IO 流的状态来同时管理多个 IO 流。先构造一个我们感兴趣的描述符（通常不止一个）的列表，然后调用一个函数，直到这些描述符中的一个已准备好进行 IO 时，该函数才返回。

通常 Unix/Linux 系统可使用 select/pselect, poll, epoll(Linux), kqueue(BSD) 满足IO多路复用功能。

select/pselect

返回值：准备就绪的描述符的数目，若超时，返回0；若出错返回-1。
具体参数说明见《Unix 环境高级编程》。但是 select 在大规模高并发的时候有如下问题：

  • select 如果任何一个 fd 返回数据，select 返回，并不告诉你那个 fd 上有数据，需要你一个个的找，一旦多了，负载自然升高。
  • select 只能最多监视（<sys/select.h>中定义的常量） FD_SETSIZE 个连接，一般是1024。
  • select 非线程安全。

poll

返回值：准备就绪的描述符的数目，若超时，返回0；若出错返回-1。

poll 没有 select 出现的问题，但是线程安全问题依然存在。

epoll

poll() 和 select() 都有诸多限制。Linux 内核在 2.5.66+ 后使用了 event poll (epoll) 组件。较为 poll() 和 select 复杂很多，但是解决了重要的性能问题，同时也新增了一些特性。poll()和select() 需要轮询文件描述符列表，检查每个文件描述符的读写状态。一旦文件描述符比较大，轮询调用列表将成为扩展的瓶颈。

epoll 将监听注册和实际监听解耦。一个系统调用初始化 epoll 上下文，另外一个从上线文增加监听文件描述符或者移除文件描述符，第三个系统调用执行事件等待。

初始化：

epoll_create1()初始化一个新的 epoll 实例并返回实例相关联的文件描述符ID（和文件没啥关系，只是 epoll 实例的操作句柄）。
flags 参数允许变化 epoll 的行为。目前只有 EPOLL_CLOEXEC 可用，允许使用 close-on-exec行为。
一旦出错，返回 -1，并设置 errno。

控制 epoll:

epoll_ctl()系统调用可以被用来往给定的 epoll 上下文中添加文件描述符和移除文件描述符。

epfd 参数为epoll_create1()创建 epoll 实例返回的文件描述符 ID。op 参数值：

  • EPOLL_CTL_ADD: 增加监听的文件描述符 fd。
  • EPOLL_CTL_DEL: 移除监听的文件描述符 fd。
  • EPOLL_CTL_MOD: 修改存在的文件描述符 fd，修改 event。

event 参数用来指定监听文件描述符的类型，多个事件可以用位与连在一起。

  • EPOLLERR 文件上出现错误。该事件总是被监听，就算没有显式指定该事件。
  • EPOLLLEF 监听文件使用边缘触发行为。默认的为水平触发。
  • EPOLLHUB 文件出现 hangup。 该事件总是被监听，就算没有显式指定该事件。
  • EPOLLIN 文件非阻塞读。
  • EPOLLONESHOT 事件一旦产生并被读取，文件自动不再被监听。
  • EPOLLOUT 文件非阻塞写。
  • EPOLLPRI 有读数据立马要紧急处理

函数成功返回 0。错误返回-1 并设置 errno。

epoll 等待事件

函数 epoll_wait() 等待文件描述符所产生事件。

函数 epoll_wait() 调用等待 timeout 参数指定的时间（单位为毫秒）。一旦返回成功，events 指针指向 epoll_event 结构的事件，文件已经准备好读和写，数目由 maxevents 参数指定。返回值为事件的数目，-1 为出错，并设置 errno 错误码，错误码具体含义可参考文档。
如果 timeout 为0，调用将会立即返回，尽管没有获取到任何事件，这种情况下会返回0。如果 timeout 为 -1，调用将一直会被阻塞，直到有事件可以获取。

边缘触发和水平触发

水平触发（LT）是默认的的工作方式，支持阻塞和非阻塞 socket。当描述符从未就绪变为就绪时，内核通过 epoll 告诉进程该描述符有事件发生，如果进程一直不对这个就绪状态不做处理，内核就一直通知，直到事件处理完成为止。 LT 方式调用的 epoll 接口就相当于速度比较快的 poll 模型。

边缘触发（ET）是高速工作方式，只处理非阻塞 socket。这种模式下，描述符从未就绪变为就绪时，内核通过 epoll 告诉进程描述符有事件发生，之后就不会再通知。

总的来说，在大并发系统中，边缘触发比水平触发更有优势。

kqueue

BSD平台使用的事件驱动机制，名称为内核事件通知机制（Kernel event notification mechanism）。提供了通用的事件监听方法，不只是可以处理 IO 事件。

1. 简介

Lighttpd (发音为”lighty”)，Lighttpd 是一个德国人领导的开源 Web 服务器软件，其根本的目的是提供一个专门针对高性能网站，安全、快速、兼容性好并且灵活的网页服务器环境。具有非常低的内存开销、CPU 占用率低、性能好以及丰富的模块等特点。Lighttpd 是众多开源轻量级的网页服务器中较为优秀的一个。支持 FastCGI、CGI、Auth、输出压缩（output compress）、URL重写、别名等重要功能。—— 维基百科

相比 Apache 和 Nginx 而言，Lighttpd 使用并不是非常广泛，但 Lighttpd 的优秀还是值得肯定的。由于 Lighttpd 源码较 Nginx 的源码更易读，而底层实现原理和 Nginx 基本一致。作为想学习 Unix/Linux 网络编程的开发人员而言，Lighttpd 的源码值得一读。
